# INDEX_KR

## 한 줄 설명
DDD, Clean Architecture, SOLID 원칙으로 코드 구조와 품질 기준을 잡아주는 스킬 모음입니다.

## 언제 쓰면 되나
- 복잡한 도메인 로직을 장기적으로 유지보수 가능하게 설계하고 싶을 때
- 비즈니스 로직과 인프라 코드를 분리하고 싶을 때
- 네이밍, 책임 분리, 함수 크기, 에러 처리 같은 규칙을 팀 기준으로 맞추고 싶을 때
- AI가 생성하는 코드도 일정한 구조 원칙을 따르게 하고 싶을 때

## 이런 상황에 특히 적합
- 서비스 규모가 커져서 코드 경계가 흐려졌을 때
- 기능은 돌아가지만 구조가 지저분해서 다음 작업이 힘들 때
- 리뷰 기준을 "취향"이 아니라 "원칙"으로 맞추고 싶을 때

## 이럴 때는 다른 폴더가 더 맞음
- DB 쿼리 최적화 중심이면 `supabase-postgres-best-practices`
- 구현 전에 아이디어 정리부터 필요하면 `brainstorming`

## 빠른 예시
- 인증/주문/결제 도메인 경계를 분리하는 설계
- use case, repository, adapter 구조 정리
- 함수와 파일이 비대해진 코드베이스 리팩터링 기준 수립

## 메모
- 즉효성 있는 "기능 추가 도구"라기보다, 코드 품질 기준을 올리는 장기적 도구에 가깝습니다.
